# Prompt 原子指令库 v1.0
> 基于融小集项目实战经验沉淀
> 适用工具：Claude Code (CC) CLI
> 最后更新：2026-05-17

---

## 使用说明

每条指令是一个**独立、可复现的最小执行单元**，称为"原子"。
- 每个原子有唯一编号（`P-XX`），可在编排指令中通过编号引用
- `{{变量}}` 表示执行时需替换的项目特定内容
- **输入依赖**：该原子依赖哪个文件或前序原子的输出
- **验收标准**：用于判断输出是否达标，不达标则追加修正指令

---

## 文件命名约定

```
输入文件:  CLAUDE.md                        （项目唯一输入源）
输出文件:  01-framework.md                  （项目推进框架）
           02-competitor-report.md           （竞争/政策分析）
           03-design-brief.md               （设计需求文档）
           04-tech-spec.md                  （技术架构方案）
           05-compliance-checklist.md       （合规/上线材料清单）
           06-external-dependencies.md      （外部依赖执行计划）
           07-store-strategy.md             （商店上架策略）
           00-index.md                      （文档索引，最后生成）
```

## 第一组：项目推进框架（P-01 系列）

### P-01 | 生成项目推进框架

**角色**：资深项目经理
**输入依赖**：`CLAUDE.md`
**输出文件**：`01-framework.md`
**预估耗时**：30-60 分钟（含 CC 生成 + 人工审核）

```
你是一个有10年互联网产品经验的资深项目经理，擅长金融/消费类APP从零到上线的全流程管理。

请读取 CLAUDE.md，基于其中的项目需求，生成一份完整的项目推进框架，输出为 01-framework.md。

文档必须包含以下章节：
1. 项目背景与定位（产品定位 + 交易结构 + 核心业务线）
2. 基于项目周期的冲刺计划（每周：目标 / 工作项 / 产出物 / 负责人）
3. MECE 准备事项清单（维度：产品/设计/技术/测试/合规/运营/商店/商务）
4. 关键路径分析（标注哪些事项是 hard dependency，延误会影响上线）
5. 风险矩阵（风险描述 / 概率 / 影响 / 应对措施）
6. 并行策略（用 ASCII 甘特图表示各模块并行关系）
7. 外部依赖优先级（P0/P1 分级 + 最晚启动时间）
8. 上线材料筹备清单（证照/协议/商店素材/备案，每项标注主体+负责人+行动）

格式要求：
- 使用 Markdown，包含清晰的二级/三级标题
- 准备事项清单用表格，含：序号/事项/维度/优先级/负责人/状态
- 甘特图用 ASCII 或 Markdown 表格模拟
- 整体不少于 600 行
```

**验收标准**：
- [ ] 上线计划每周有明确产出物（不是泛泛的"推进开发"）
- [ ] MECE 清单覆盖 8 个维度，每维度至少 5 条
- [ ] 甘特图能体现至少 3 个模块的并行关系
- [ ] 风险矩阵至少 6 条，含合规风险

---

### P-01-FIX | 框架修正（迭代用）

**输入依赖**：`01-framework.md` + 具体修改意见
**使用时机**：P-01 输出不满足验收标准时

```
请读取 01-framework.md，按以下修改意见进行调整，覆盖保存：

{{修改意见，例如：
- 第三章 MECE 清单缺少"商务"维度，请补充
- 第四章关键路径未标注三方接口联调的依赖关系，请补充
- 风险矩阵中的"合规风险"描述过于笼统，请细化为具体监管条款}}

修改后请在文档末尾追加"变更日志"章节，记录本次修改内容和日期。
```

## 第二组：竞争与政策分析（P-02 系列）

### P-02 | 生成竞争/政策分析报告

**角色**：行业研究分析师 + 合规专家
**输入依赖**：`CLAUDE.md`
**输出文件**：`02-competitor-report.md`
**注意**：此原子需要联网搜索，确保 CC 有 web search 权限

```
你是一个资深行业研究分析师，同时具备金融监管合规背景。

请读取 CLAUDE.md 中的产品定位和竞品信息，生成竞争与政策分析报告，输出为 02-competitor-report.md。

报告必须包含：
1. 竞品功能矩阵（竞品名称 / 核心功能 / 目标用户 / 差异点）
2. 竞品 UI 风格分析（主色调 / 交互密度 / 信息架构特点）
3. 行业政策动态（搜索最新的 {{行业}} 监管政策，聚焦近12个月）
4. 政策对本项目的影响分析（逐条对应，标注：影响程度 / 应对方向）
5. 合规红线清单（本项目必须满足的监管要求，按优先级排序）
6. 机构应对建议（同类产品如何处理合规问题，可参考案例）

格式要求：
- 竞品矩阵用表格
- 政策影响分析用"政策原文摘要 → 影响判断 → 建议行动"三列结构
- 合规红线用醒目标注（每条加 ⚠️ 前缀）
```

**验收标准**：
- [ ] 竞品矩阵至少覆盖 CLAUDE.md 中提到的所有竞品
- [ ] 政策动态引用来源可追溯（注明出处和日期）
- [ ] 合规红线与本项目的业务模式直接相关（无泛泛而谈）

## 第三组：设计需求文档（P-03 系列）

### P-03 | 生成设计需求文档

**角色**：资深产品设计负责人
**输入依赖**：`01-framework.md` + `CLAUDE.md`
**输出文件**：`03-design-brief.md`

```
你是一个资深产品设计负责人，负责向 UI 设计师交付可直接执行的设计需求文档。

请读取 CLAUDE.md 和 01-framework.md，生成设计需求文档，输出为 03-design-brief.md。
设计师应能不需要额外沟通，直接基于此文档开始 Figma 设计。

文档必须包含：
1. 页面清单（编号 / 页面名称 / 功能说明 / 关键 UI 元素 / 交互要点）
2. 设计风格规范
   - 主色调 + 辅色（给出色值参考范围）
   - 字体方案（标题/正文/数字/标签）
   - 视觉密度（轻量/标准/信息密集）
   - 图标风格（线性/面性/混合）
3. 竞品风格差异对比表（参考竞品 / 相似之处 / 本项目差异点 / 原因）
4. 公共组件体系（按钮 / 弹窗 / 卡片 / 表单 / 加载态 / 空状态）
5. 应用商店视觉素材清单（截图数量 / 尺寸要求 / 内容主题）
6. 四周设计排期（每周交付哪些页面/组件，与 01-framework.md 对齐）
7. 交付规范（切图命名 / 标注工具 / 文件格式 / 提交方式）
8. 设计注意事项（品牌禁忌 / 合规展示要求 / 特殊机型适配）

格式要求：
- 页面清单用表格，按功能模块分组（如：首页模块 / 借款流程 / 个人中心）
- 总页面数量与 CLAUDE.md 中描述的业务范围匹配
```

**验收标准**：
- [ ] 页面清单覆盖完整用户旅程（从首次打开到核心业务完成）
- [ ] 色值方案与竞品有明显差异（不能和竞品撞色）
- [ ] 四周排期可执行（工作量分布合理，不头重脚轻）

## 第四组：技术架构方案（P-04 系列）

### P-04 | 生成技术架构方案

**角色**：资深全栈架构师
**输入依赖**：`CLAUDE.md` + `01-framework.md`
**输出文件**：`04-tech-spec.md`

```
你是一个资深全栈架构师，负责为移动端 APP 项目制定技术方案。

请读取 CLAUDE.md 和 01-framework.md，生成技术架构方案，输出为 04-tech-spec.md。

文档必须包含：
1. 技术选型（前端框架 / 后端语言 / 数据库 / 缓存 / 消息队列），每项说明选择理由
2. 系统架构图（用 ASCII 或 Mermaid 表示主要模块和数据流向）
3. 数据库 ER 图（核心表结构，用 Mermaid erDiagram 语法）
4. API 接口清单（模块 / 接口名 / Method / 路径 / 请求参数 / 返回结构 / 备注）
5. 三方服务集成清单（服务名称 / 用途 / 对接方式 / 准入要求 / 预计周期）
6. Mock 方案（哪些接口需要 Mock / 推荐工具 / Mock 数据维护方式）
7. 安全方案（数据加密 / 接口鉴权 / 防刷策略 / 合规数据处理）
8. 部署方案（环境划分 / CI/CD 建议 / 发布流程）

格式要求：
- API 清单用表格，按业务模块分组
- ER 图使用 Mermaid 代码块
- 三方集成清单标注哪些是 P0（上线必须）/ P1（可延后）
```

**验收标准**：
- [ ] API 清单覆盖所有页面清单（03-design-brief.md）中的交互场景
- [ ] 三方集成清单与 CLAUDE.md 中的外部依赖对齐
- [ ] ER 图能被开发直接用于建表（字段类型明确）

## 第五组：合规与上线材料（P-05 系列）

### P-05 | 生成合规与上线材料清单

**角色**：合规专员 + 运营经理
**输入依赖**：`CLAUDE.md` + `02-competitor-report.md` + `01-framework.md`
**输出文件**：`05-compliance-checklist.md`

```
你是一个负责互联网金融产品上线的合规专员，同时负责应用商店审核流程。

请读取 CLAUDE.md、02-competitor-report.md 和 01-framework.md，
生成合规与上线材料清单，输出为 05-compliance-checklist.md。

文档必须包含：
1. 证照清单（所需资质 / 持证主体 / 当前状态 / 获取周期 / 负责人）
2. 协议文本清单（协议名称 / 展示位置 / 法律要求来源 / 起草责任方）
3. 隐私合规要点（数据收集项 / 使用目的 / 存储方式 / 用户授权方式）
4. 金融营销合规要求（禁止用语 / 必须展示内容 / 风险提示规范）
5. 应用商店材料清单（各渠道：材料项 / 规格要求 / 审核重点 / 预计周期）
6. 备案清单（ICP备案 / 等保 / 金融相关备案，每项含状态和负责人）
7. 上线前检查 Checklist（按维度：产品/技术/合规/运营，共 30+ 条）

格式要求：
- 所有清单用表格
- 每条加状态列：待启动 / 进行中 / 已完成
- 商店审核周期用表格对比：iOS / 华为 / 小米 / OPPO / vivo / 应用宝
```

**验收标准**：
- [ ] 证照清单与 CLAUDE.md 中的业务模式匹配（不遗漏监管要求）
- [ ] 商店材料清单包含所有主流安卓渠道
- [ ] 上线前 Checklist 可直接用于上线评审会

## 第六组：外部依赖执行计划（P-06 系列）

### P-06 | 生成外部依赖执行计划

**角色**：项目经理
**输入依赖**：`CLAUDE.md` + `01-framework.md` + `04-tech-spec.md`
**输出文件**：`06-external-dependencies.md`

```
你是一个擅长多方协调的项目经理。

请读取 CLAUDE.md、01-framework.md 和 04-tech-spec.md，
梳理所有外部依赖，生成执行计划，输出为 06-external-dependencies.md。

文档必须包含：
1. 外部依赖总览（依赖项 / 类型 / 优先级 / 影响范围 / 当前状态）
2. P0 依赖执行计划（最晚启动时间 / 对接负责人 / 具体行动项 / 里程碑）
3. P1 依赖执行计划（同上格式）
4. 依赖关系图（哪些依赖之间有先后顺序，用 ASCII 树状图表示）
5. 风险预案（如果某个 P0 依赖延误，备选方案是什么）
6. 周度跟进模板（用于项目周例会的依赖跟进表格格式）

优先级定义：
- P0：上线必须，延误直接影响上线时间
- P1：上线建议，可以 V1.1 补充

格式要求：
- 每个依赖项的"具体行动项"要拆解到人和日期
- 风险预案写出具体的替代方案，不能只写"提前沟通"
```

## 第八组：文档质检与索引（P-08 系列）

### P-08 | 质检所有输出文档

**角色**：项目质检
**输入依赖**：`CLAUDE.md` + 所有已生成文档（01~07）
**输出文件**：`00-index.md`（含质检报告）
**使用时机**：所有文档生成完毕后执行

```
请执行以下质检任务，并将结果输出为 00-index.md：

【第一部分：文档索引】
列出所有已生成文档（01~07），每个文档包含：
- 文件名
- 文档标题
- 核心内容摘要（3句话以内）
- 字数统计
- 最后更新时间

【第二部分：一致性检查】
对照 CLAUDE.md 的原始需求，逐项检查：

1. 产品定位是否在所有文档中保持一致？
2. 上线时间线是否在各文档（框架/设计排期/外部依赖/商店策略）中对齐？
3. 外部依赖清单：CLAUDE.md 中列出的依赖是否全部出现在 06-external-dependencies.md？
4. 竞品名称：CLAUDE.md 中的竞品是否全部出现在 02-competitor-report.md 和 03-design-brief.md？
5. 技术三方服务：04-tech-spec.md 中的三方集成是否全部出现在 05-compliance-checklist.md？

【第三部分：差异报告】
对于上述检查中发现的不一致项，列出：
- 不一致描述
- 涉及文档
- 建议修正方式

【第四部分：待人工确认项】
列出所有文档中出现的"待定/TBD/需确认"标记，汇总成一张表，
供 PM 在第一次团队对齐会上逐一确认。
```

**验收标准**：
- [ ] 差异报告为空，或每条差异都有明确修正建议
- [ ] 待人工确认项不超过 10 条（超过说明 CLAUDE.md 质量需提升）

## 主编排指令（Orchestrator）

### ORCH-01 | 一键启动完整文档生成

**使用前提**：`CLAUDE.md` 已完成，位于项目根目录
**执行方式**：在 CC CLI 中直接粘贴运行

```
请按以下顺序依次执行，每步完成后确认文件已生成再执行下一步：

步骤1：执行 P-01，生成 01-framework.md
步骤2：执行 P-02，生成 02-competitor-report.md（与步骤1并行）
步骤3：执行 P-03，生成 03-design-brief.md（依赖步骤1完成）
步骤4：执行 P-04，生成 04-tech-spec.md（依赖步骤1完成）
步骤5：执行 P-05，生成 05-compliance-checklist.md（依赖步骤2和步骤1完成）
步骤6：执行 P-06，生成 06-external-dependencies.md（依赖步骤1和步骤4完成）
步骤7：执行 P-07，生成 07-store-strategy.md（依赖步骤5完成）
步骤8：执行 P-08，生成 00-index.md 并输出质检报告

并行策略：步骤1和步骤2可同时进行；步骤3和步骤4可在步骤1完成后同时进行。
每步完成后在终端输出"✅ [文件名] 已生成，字数：XXX"。
```
